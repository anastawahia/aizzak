## 0. كيف تُقرأ هذه الوثيقة

### 0.1 مبدأ حاكم — القرارات محسومة
المعمارية وحزمة التقنيات في هذه الوثيقة **قرارات نهائية اتّخذها العميل**. دور المورّد هو **التنفيذ المطابق**، لا إعادة الاختيار. أي انحراف عن القيود المُعلَّمة بـ«يجب» يتطلّب **موافقة خطّية مسبقة** من العميل، وإلا يُعدّ مخالفة قابلة للرفض عند التسليم.

### 0.2 كلمات الإلزام (RFC 2119)
| الكلمة | المعنى |
|--------|--------|
| **يجب (MUST)** | إلزامي مطلق. مخالفته ترفض التسليم. |
| **ينبغي (SHOULD)** | موصى به بقوّة؛ الانحراف يحتاج تبريراً موثّقاً ومقبولاً. |
| **يمكن (MAY)** | خيار مسموح لتقدير المورّد. |

### 0.3 ترميز المتطلبات والتتبّع
- كل متطلّب قابل للاختبار يحمل معرّفاً: `ARC‑` (معماري) · `FR‑` (وظيفي) · `NFR‑` (غير وظيفي) · `SEC‑` (أمني) · `DAT‑` (بيانات) · `API‑` · `EVT‑` (أحداث) · `OPS‑` (نشر/تشغيل) · `OQ‑` (نقطة قرار مفتوحة).
- عمود **«المصدر»** يربط المتطلّب بقرار التصميم الحاكم (`D‑xx` / `DD‑xx` / `I‑x`) لأغراض التتبّع.

## 1. الملخّص التنفيذي

المطلوب بناء **Backend بلغة Python** لمنصّة ذكاء اصطناعي متعددة الوكلاء (Multi‑Agent AI Platform)، **قابلة للتوسّع لخدمة آلاف المستخدمين المتزامنين**.

المنصّة تُقدّم لكل مستخدم مساحة عمل (Workspace) واحدة تضمّ مجموعة **وكلاء (Agents)** ذكية متخصّصة، تشترك في ملفات المستخدم، وتستطيع العمل منفردة أو ضمن **سلاسل تنفيذ متعددة الوكلاء (Multi‑Agent Workflows)**. النظام مبني ليكون **قابلاً للتوسعة عبر الإضافات (Plugin Architecture)**: يمكن إضافة وكيل جديد بمجرّد وضعه في مجلّد الوكلاء دون تعديل النواة.

**الوكلاء المُسبقة الخمسة (Pre‑built Agents):**
1. **RAG Agent** — استرجاع معزّز بالتوليد فوق مستندات المستخدم.
2. **Data Analysis Agent** — تحليل البيانات.
3. **Image Generation Agent** — توليد الصور.
4. **Video Generation Agent** — توليد الفيديو.
5. **File Editing Agent** — تحرير الملفات.

**النمط المعماري:** Modular Monolith، وداخل كل وحدة (Module) تُطبَّق Hexagonal Architecture (Ports & Adapters)، مع Plugin Architecture للوكلاء، و Event‑Driven Architecture **للعمليات الثقيلة فقط** عبر Redis Streams.

## 2. النطاق (Scope)

### 2.1 ضمن النطاق (In Scope)
- خدمة Backend كاملة (API + منطق أعمال + معالجة خلفية) وفق المعمارية المحدّدة.
- الوكلاء الخمسة المُسبقة + إطار الإضافات (Plugin) لدعم وكلاء مستقبليين.
- **عشر وحدات أعمال في v1** — بما فيها وحدتان مُرقّيتان: `integrations` (موصّلات/OAuth/MCP) و `usage` (قياس/حصص/حدود) — انظر `FR‑120…124` و `FR‑130…134`.
- إطار الأدوات (Tool System)، وإطار سلاسل التنفيذ (Workflow Engine).
- طبقة تجريد المزوّدين (LLM / Embedding / Vector Store / Storage / Cache) عبر Ports و Adapters.
- المصادقة (Firebase) والتفويض (RBAC) وعزل المستأجرين (Tenant Isolation).
- مخطط قاعدة البيانات وهجرات Alembic لكل وحدة.
- عقود الـAPI والأحداث (Contracts) وتطبيقها.
- سكربتات/إعدادات النشر (Nginx · Uvicorn · PgBouncer · Redis · MinIO).
- الاختبارات (وحدة/تكامل) وفرض الطبقات آلياً (import‑linter).

### 2.2 خارج النطاق (Out of Scope)
- أي واجهة أمامية (Frontend / Mobile).
- **Observability التشغيلية** (Audit · Metrics · Tracing): **لا تُنفَّذ الآن**، لكن **يجب** ترك نقاط توسعة تسمح بإضافتها لاحقاً **دون إعادة هيكلة** (القسم 12 — `NFR‑OBS`).
- التعاون متعدّد المستخدمين داخل Workspace واحد (نموذج البيانات يدعمه مستقبلاً؛ لا يُفعَّل في v1 — انظر `I‑1`).
- الأدوار المخصّصة (Custom Roles): الأدوار ثابتة في v1، والتخصيص نقطة توسعة.
- **الفوترة المالية وبوّابات الدفع/الفواتير**: خارج v1؛ وحدة `usage` تقتصر على **القياس والحصص والميزانيات والحدود** (`FR‑130…134`)، والدفع الفعلي نقطة توسعة (بلا خدمات جديدة).
- أي بنية تحتية غير مذكورة في القسم 5 — **إضافتها ممنوعة دون موافقة** (القسم 5.3).

## 3. مسرد المصطلحات (Glossary)

| المصطلح | التعريف |
|---------|---------|
| **Workspace** | مستأجر شخصي واحد لكل مستخدم؛ يحوي وكلاءه وملفاته ومحادثاته. هو حدّ عزل البيانات. |
| **Agent** | مكوّن ذكي عديم الحالة (Stateless) يُنشأ لكل طلب، يحمل بياناته من التخزين، وينسّق تنفيذ مهمة، ثم يُتلَف. |
| **Module** | وحدة أعمال مستقلّة مغلقة الحدود، تطبّق Hexagonal داخلياً وتملك Schema خاصاً في قاعدة البيانات. |
| **Port** | واجهة (Interface/Protocol) يعرّفها النطاق ولا يعرف تنفيذها. |
| **Adapter** | تنفيذ ملموس لـ Port يتكامل مع تقنية خارجية (Postgres/Redis/MinIO/مزوّد ذكاء اصطناعي). |
| **Plugin** | وكيل قابل للاكتشاف والتسجيل آلياً عبر PluginLoader و AgentRegistry. |
| **Workflow** | سلسلة تنفيذ تنسّق عدة وكلاء (مثل Planner → RAG → Image → Reviewer). |
| **Tool** | قدرة قابلة للاستدعاء من الوكيل عبر عقد موحّد دون معرفة تنفيذها الداخلي. |
| **ExecutionContext** | سياق الطلب الحامل لهوية المستخدم و `workspace_id` و `correlation_id`. |

## 4. القيود المعمارية الإلزامية (Mandatory Architectural Constraints)

> هذا القسم هو **جوهر الوثيقة**. جميع بنوده **إلزامية**.

### 4.1 مبادئ التصميم
`ARC‑01` **يجب** الالتزام بـ: SOLID · High Cohesion · Low Coupling · Separation of Concerns · Dependency Inversion · Interface Segregation · Domain‑Driven Design (قدر الإمكان). *(المصدر: المتطلبات الأصلية)*

### 4.2 النمط المعماري العام
`ARC‑02` **يجب** أن يكون النظام **Modular Monolith** بوحدات مستقلّة، ووكلاء مستقلّين، و Framework مستقل، و Infrastructure معزولة.

`ARC‑03` **يجب** تطبيق **Hexagonal Architecture** داخل كل Module، بحيث يحوي كل Module: `Domain` · `Application` · `Ports` · `Adapters` · (Infrastructure خاصة إن لزم).

`ARC‑04` **يجب** ألّا يعتمد الـ`Domain` على أي Framework أو Database أو FastAPI أو أي تقنية بنية تحتية — إطلاقاً.

### 4.3 الطبقات الخمس (بالضبط، لا زيادة)
`ARC‑05` **يجب** أن يتكوّن التطبيق من **خمس طبقات حصراً**، وأي طبقة إضافية تحتاج موافقة العميل:

```
Application
├── Framework Layer        (نواة المنصّة: التمهيد، الإعداد، DI، دورة الحياة)
├── API Layer              (استقبال طلبات FastAPI، WebSocket، التوثيق)
├── Agents Layer           (الوكلاء الذين ينسّقون التنفيذ)
├── Business Modules Layer  (منطق الأعمال والقدرات الأساسية)
└── Infrastructure Layer   (Adapters: Postgres/Redis/MinIO/Firebase/المزوّدون)
```

### 4.4 فئات المكوّنات الخمس
`ARC‑06` **يجب** تصنيف المكوّنات ضمن خمس فئات: **API Components** · **Platform Components** (PluginLoader · Registry · Workflow · Event Bus …) · **Agent Components** · **Business Components (Modules)** · **Infrastructure Components (Adapters)**.

### 4.5 قواعد التواصل بين الوحدات
`ARC‑07` **يمنع منعاً باتّاً** أن تستدعي وحدة وحدةً أخرى مباشرة (`Module A → Module B`).

`ARC‑08` **يجب** أن يتمّ التواصل بين الوحدات **حصراً** عبر أحد: `Ports` / `Interfaces` / `Domain Events` (داخل الوحدة) / `Global Events` (بين الوحدات، عبر Redis Streams) — بحسب نوع العملية.

### 4.6 قواعد التبعية بين الطبقات (Dependency Rules)
`ARC‑09` اتجاه التبعية **يجب** أن يكون نحو الداخل (Dependency Inversion): تعتمد الطبقات الخارجية على تجريدات النطاق، لا العكس. تحديداً:
- `Domain` لا يستورد شيئاً من `Application` / `Adapters` / `Infrastructure` / أي مكتبة إطار.
- `Application` يعتمد على `Domain` و `Ports` فقط، ولا يعرف تنفيذ الـAdapters.
- `Adapters/Infrastructure` تنفّذ الـPorts وتُحقن في زمن التشغيل.
- `API Layer` لا يحوي منطق أعمال؛ يفوّض إلى `Application`/`Agents`.

`ARC‑10` **يجب** فرض هذه الحدود **آلياً** عبر **`import‑linter`** ضمن خط التكامل المستمر (CI)، بحيث يفشل البناء عند أي مخالفة استيراد. *(المصدر: D‑17, DD‑14)*

### 4.7 حقن التبعيات
`ARC‑11` **يجب** استخدام **Manual Dependency Injection** فقط. **يُمنع** استخدام أي DI Container خارجي إلا بموافقة صريحة من العميل. *(المصدر: المتطلبات الأصلية)*

### 4.8 الإعداد المركزي
`ARC‑12` **يجب** إنشاء **Settings Service مركزي**؛ و**يُمنع** على أي Module قراءة `.env` مباشرة. المصدر الوحيد للإعداد هو عقد `Settings` المقدَّم من طبقة `infrastructure/config`. *(المصدر: DD‑11)*

### 6.14 الاستخدام والحصص (Usage & Quotas) — وحدة v1
> **رُقّيت** من «محجوزة» إلى **وحدة v1 مُنفَّذة** (schema `usage`). نطاق v1: **قياس + حصص + ميزانيات + حدود**. بوّابات الدفع/الفوترة المالية **خارج v1** (نقطة توسعة، بلا خدمات جديدة).

| المعرّف | المتطلّب | المصدر |
|---------|----------|--------|
| `FR‑130` | **يجب** تنفيذ وحدة `usage` تسجّل **استهلاك الرموز/التكلفة** لكل (`workspace` · `agent` · مزوّد)، وتوفّر **حصصاً وميزانيات وحدوداً** قابلة للإعداد. | v1 (رُقّيت من 12.1) |
| `FR‑131` | **يجب** التقاط الاستهلاك (بعد العملية) دون خرق حدود الوحدات عبر **منفذ وارد (Inbound Port)** لوحدة `usage` يستدعيه المُستدعي **متزامناً** فيُلحِق قيداً في **سجلّ استهلاك داخلي (append‑only)** — **لا عبر Redis Streams/Global Events** (اتّساقاً مع قصر `EVT‑10` على العمليات الثقيلة). يبقى الإلحاق خفيفاً على المسار الساخن، والتجميع/التدوير (rollup) شأن داخلي للوحدة. | ARC‑07/08 · EVT‑10 |
| `FR‑132` | **يجب** دعم **فرض الحدّ** (قبل العملية) عبر **Inbound Port** يستعلمه المستدعي فيُعيد **كائن قرار (Decision Object)** — لا `bool` مجرّداً — بسماح/رفض عند تجاوز الحصّة أو الميزانية. **يجب** أن يُصمَّم عقد المنفذ ليقبل التطوّر لاحقاً إلى **حجز/تسوية ذرّيَّين (reserve/commit)** (بإضافة `reservation_id` وأسلوبَي حجز ثمّ تأكيد) **دون كسر العقد** — نقطة توسعة (القسم 12) تحسّباً لتشديد الفرض ضدّ سباق «الفحص‑ثمّ‑الاستهلاك». **يُستدعى فرض الحدّ (والالتقاط في `FR‑131`) من طبقة الوكلاء (المُنسِّق) حصراً**، لا من داخل وحدة أعمال أخرى — تفادياً لاقتران مباشر بين الوحدات؛ وتعبر الاستدعاءات الحدود عبر **منافذ `usage` الواردة** فقط. | جديد · ARC‑07/08 · القسم 12 |
| `FR‑133` | تُواءم القيم الرقمية (الحصص/الحدود) مع جداول SLO المعتمدة (`OQ‑02`). | OQ‑02 |
| `FR‑134` | **يجب** أن يحمل كل قيد استهلاك **مفتاح إدمپوتنسي طبيعياً** (مثل `operation_id` أو `(agent_run_id, provider, seq)`) مفروضاً بـ`UNIQUE` داخل schema `usage`؛ فإعادة استدعاء منفذ الالتقاط بنفس المفتاح **لا تُضاعف العدّ** (تُتجاهَل بهدوء). مصدر القيد هو **المُنسِّق** (طبقة الوكلاء) الذي يوفّر (`workspace_id · agent · provider · tokens · cost · operation_id`). | جديد · FR‑131 |

## 12. نقاط التوسعة المستقبلية (Extension Points)

**يجب** أن يترك التصميم نقاط التوسعة الآتية **دون إعادة هيكلة لاحقاً**:
- **Observability**: `correlation_id`/`causation_id` مضمّنان أصلاً في المظروف والأخطاء لتفعيل Tracing/Audit/Metrics لاحقاً (`NFR‑OBS`).
- **وكلاء جدد**: عبر Plugin دون تعديل النواة (`FR‑30…32`).
- **وحدات جديدة**: مجموعة الوحدات مفتوحة للتوسّع عبر القالب السداسي القياسي دون تعديل الوحدات القائمة (`FR‑110…112`)؛ الثلاث المرشّحات المتبقّية محجوزة في **القسم 12.1**.
- **مزوّدون جدد**: عبر Adapters خلف المنافذ المحايدة (`FR‑70…73`).
- **تعاون متعدّد المستخدمين**: نموذج `access` عام (`RoleAssignment: user × workspace × role`) يتيح التعاون مستقبلاً دون تغيير مخطط (`I‑1`).
- **أدوار مخصّصة**: توسعة فوق كتالوج الأدوار الثابت (`DD‑10`).
- **تشديد فرض الحصص**: منفذ `usage` الوارد يعيد **كائن قرار** قابلاً للتطوّر إلى **حجز/تسوية ذرّيَّين (reserve/commit)** دون كسر العقد (`FR‑132`).
- **إصدارات API/أحداث جديدة**: `/api/v2` و `.vN+1` بآليّات مُعرَّفة (`API‑05`, `EVT‑02`).

---

### 12.1 الوحدات المستقبلية المحجوزة (Reserved Future Modules)

> **محجوزة لا مُلزَمة:** الوحدات الثلاث أدناه **لا تُبنى في v1** — بعد ترقية **Integrations** و**Usage** إلى وحدتَي v1 (القسمان 6.13 و 6.14) — لكن التصميم **يحجز لها مقعداً** (اسم Schema + نمط منفذ + مجرى أحداث) لتُضاف لاحقاً **دون إعادة هيكلة** (`FR‑110…112`). كل وكيل جديد غالباً يعتمد على واحدة أو أكثر منها.

| الوحدة | Schema محجوز | الغرض | تُمكّن من الوكلاء | الأولوية |
|--------|--------------|-------|------------------|----------|
| **Scheduling / Triggers** | `scheduling` | جدولة Cron · Webhooks · تشغيل خلفي/مستقل للوكلاء | الوكلاء المجدوَلون/المستقلّون · Supervisor | 🔴 عالية |
| **Sandbox / Code Execution** | `sandbox` | بيئة تنفيذ كود معزولة ومؤقتة | Coding/Code‑Interpreter · Data Analysis · Computer‑Use | 🟡 متوسطة |
| **Runs / Evaluation** (Agent‑Ops) | `runs` | سجلّ تشغيل دائم · Traces · تقييمات · تغذية راجعة (يبني على `NFR‑OBS`) | الجودة والمراقبة لكل الوكلاء | 🟡 متوسطة |

> ⚠️ **تنبيه `ARC‑15`:** وحدة **Sandbox** قد تتطلّب خدمة بنية تحتية جديدة (زمن تشغيل تنفيذ معزول مثل gVisor/Firecracker/حاويات مؤقتة) خارج حزمة القسم 5 الحالية — إدراجها يخضع لموافقة خطّية وقت البناء. بقية الوحدتين (Scheduling · Runs) تعملان ضمن الحزمة المعتمدة (Postgres · Redis · Nginx) بلا خدمات جديدة.

## 15. معايير القبول (Acceptance Criteria / Definition of Done)

> يُقبل التسليم فقط عند تحقّق **كل** ما يلي، ويُعدّ إخفاق أيٍّ منها سبباً للرفض.

| المعرّف | المعيار |
|---------|---------|
| `AC‑01` | **مطابقة الطبقات**: `import‑linter` يمرّ في CI دون أي مخالفة؛ لا استيراد يخرق حدود `ARC‑09/10`. |
| `AC‑02` | **نقاء النطاق**: لا وجود لأي استيراد لـFastAPI/DB/Framework داخل أي `Domain` (`ARC‑04`). |
| `AC‑03` | **عزل الوحدات**: لا استدعاء مباشر `Module A → Module B`؛ التواصل فقط عبر Ports/Events (`ARC‑07/08`)، ولا FK عابر بين schemas الوحدات (`DAT‑02`). |
| `AC‑04` | **إثبات الـPlugin**: عرض حيّ لإضافة وكيل جديد وحذفه **دون تعديل النواة**، واكتشافه آلياً عبر PluginLoader/AgentRegistry (`FR‑30…32`). |
| `AC‑05` | **عزل المستأجر مُثبَت**: اختبارات تُظهر منع أي تسرّب بيانات عبر الـworkspaces على طبقتَي RLS + الترشيح التطبيقي (`SEC‑03/04`)، مع دور DB خاضع لـRLS. |
| `AC‑06` | **دورة حياة الوكيل**: الحالات الخمس + مسار الفشل مطبّقة ومختبَرة (`FR‑15`)، والوكيل عديم الحالة يُنشأ لكل طلب ويُتلَف (`FR‑10/11`). |
| `AC‑07` | **مطابقة عقد الـAPI**: أخطاء RFC 9457، ترقيم بالمؤشّر، snake_case، مسار `/api/v1` (`API‑01…05`). |
| `AC‑08` | **مطابقة عقد الأحداث**: مظاريف CloudEvents صحيحة، تسمية `<module>.<aggregate>.<event>.vN`، JSON Schemas موجودة، و**Idempotency فعّالة** (لا أثر مزدوج عند إعادة التسليم) (`EVT‑01…05`). |
| `AC‑09` | **جودة الكود**: `Ruff` نظيف، و`mypy --strict` يمرّ بلا أخطاء، واختبارات `pytest` (+ asyncio + testcontainers) خضراء (`DD‑14`). |
| `AC‑10` | **الأسرار**: لا أسرار في الكود/`.env` المكشوف؛ كل الأسرار عبر Vault، ومفاتيح المستخدمين مُعمّاة عبر Transit (`SEC‑05`, `FR‑75`). |
| `AC‑11` | **البثّ**: WebSocket و Streaming Responses يعملان (`FR‑90/91`). |
| `AC‑12` | **Workflow**: تشغيل سلسلة متعددة الوكلاء (مثال `Planner→RAG→Image→Reviewer`) قابل للتوسعة (`FR‑40/41`). |
| `AC‑13` | **حسم `OQ‑01…06`**: جميع نقاط القرار المفتوحة مُوقَّعة ومطبّقة قبل التسليم النهائي. |
| `AC‑14` | **قابلية توسّع الوحدات**: إثبات موثّق (دليل + مثال مرجعي) لإضافة وحدة أعمال جديدة عبر القالب القياسي دون تعديل الوحدات القائمة، مع مرور `import‑linter` (`FR‑110…112`). |
| `AC‑15` | **وحدة `integrations`**: عرض حيّ لربط موصّل عبر OAuth، تخزين أسراره **مُعمّاة (Vault Transit)** لا بنصّ صريح، واستهلاكه من وكيل **عبر نظام الأدوات** دون اقتران مباشر، مع **تجديد رمز كسول** (`FR‑120…124`). |
| `AC‑16` | **وحدة `usage`**: اختبار يُظهر **فرض الحصّة/الميزانية** (سماح/رفض) قبل العملية عبر المنفذ الوارد، و**التقاط استهلاك إدمپوتنسي** (لا عدّ مزدوج عند إعادة الاستدعاء) **دون أحداث Redis Streams**، والتزام حدود الوحدة (`FR‑130…134` · `DAT‑02`). |
