# معايير الكود (Code Standards)

> تُفرَض آلياً حيث أمكن: `Ruff` (تنسيق + فحص) · `mypy --strict` · `import-linter` (`DD‑14`, D‑17).
> الهدف: كود يقرأ كأنه كُتب بيد واحدة، ويحترم الطبقات والحدود دون فحص بشري.

## 1) اللغة والأدوات
- Python **3.12+**. تلميحات أنواع **إلزامية** على كل دالة عامة (`mypy --strict`, بلا `Any` ضمنية).
- التنسيق والفحص عبر `Ruff` (سطر 100، اقتباس مزدوج). لا تنسيق يدوي مخالف.
- التبعيات في `pyproject.toml` فقط؛ لا تثبيت خارجه.
- كل I/O **async**؛ لا استدعاء متزامن حاجب داخل مسار async (استخدم مِنفذاً غير متزامن أو `run_in_executor` عند الضرورة القصوى).
- **DTO والإعداد عبر Pydantic v2 حصراً** (`ARC‑14`)؛ لا نماذج تحقّق/إعداد بديلة (يبقى النطاق نقياً بلا Pydantic — بند 3).

## 2) التسمية والبنية
| العنصر | الاصطلاح |
|--------|----------|
| module/package | `snake_case` |
| class | `PascalCase` |
| function/var | `snake_case` |
| constant | `UPPER_SNAKE` |
| Port (Protocol) | اسم القدرة: `LLMProvider`, `FileRepository` |
| Adapter | التقنية + الدور: `QdrantVectorStore`, `SqlFileRepository` |
| Use-Case | فعل + مفعول: `RegisterUpload`, `AssignRole` |
| Domain Event | ماضٍ: `FileUploaded`, `RoleAssigned` |
| ملف الاختبار | `test_<unit>.py` |

## 3) قواعد الطبقات (تُطابق `.importlinter`)
- **Domain نقي:** dataclasses/كائنات نقية، بلا FastAPI/SQLAlchemy/Redis/Pydantic/infra. المنطق والثوابت فقط.
- **Application:** Use-Cases تنسّق النطاق عبر **منافذ محقونة**؛ لا تعرف تقنية.
- **Ports:** `typing.Protocol` للمُقادة، `ABC` للعقود القابلة للوراثة (`BaseAgent`/`BaseTool`).
- **Adapters:** المكان **الوحيد** لاستيراد مكتبات التقنية؛ تنفّذ منفذاً وتترجم أخطاءه إلى أخطاء نطاق/تطبيق.
- **API:** رقيقة — تحقّق DTO، تفويض، استدعاء Use-Case/Agent، تحويل إلى استجابة. **لا منطق أعمال.**
- **لا وحدة تستورد أخرى**؛ التواصل عبر Inbound Port محقون أو حدث.

## 4) حقن التبعيات (Manual DI)
- **Composition Root الوحيد** (`framework/di/composition_root.py`) يبني المحوّلات ويربطها بالمنافذ.
- بقية الكود يتلقّى تبعياته عبر **المُنشئ/المعاملات** — لا `import` لمحوّل محسوس، لا Singletons عامة، لا Service Locator.
- `ExecutionContext` يُحقن request/job-scoped ويُمرَّر صراحةً (لا سياق عام في thread-local).

## 5) معالجة الأخطاء
- النطاق يرفع أخطاء نطاق نظيفة (`DomainError` وفروعها) بلا رموز HTTP.
- المحوّلات تترجم أخطاء التقنية إلى أخطاء تطبيق ذات معنى (لا تسريب `asyncpg.Error` للأعلى).
- طبقة API تحوّل إلى **RFC 9457** عبر معالج موحّد؛ كل خطأ يحمل `code` ثابتاً من الكتالوج (`03-api-spec.md`) و`correlation_id`.
- لا `except:` عارية؛ لا ابتلاع صامت. `500` لا يسرّب تفاصيل داخلية.

## 6) قاعدة البيانات والمعاملات
- الوصول عبر **Repository** فقط؛ لا SQL في التطبيق/النطاق.
- كل استعلام مستأجَر يضبط `app.workspace_id` (RLS) **ويضيف** `WHERE workspace_id` (دفاع بعمق).
- **حدّ المعاملة = Use-Case واحد**؛ أثر النطاق + صف Outbox في نفس المعاملة (لا نشر مباشر).
- القفل التفاؤلي على `version` للتعديلات؛ تعارض ⇒ `common.conflict` (409).

## 7) الأحداث والعمّال
- لا نشر مباشر إلى Redis من الوحدات — **اكتب Outbox**؛ المُرحّل ينشر.
- المستهلك **idempotent** عبر `processed_events` داخل معاملة الأثر.
- العامل يضبط سياق المستأجر قبل أي كتابة نيابةً عن `workspace_id`.

## 8) الوكلاء والأدوات (Stateless)
- الوكيل **بلا حالة في RAM** بين الطلبات؛ يُنشأ لكل طلب، يحمّل ما يلزم، ثم يُتلَف (دورة الحياة `FR‑15` · D‑05/D‑13).
- الوكيل يستخدم Tools بالاسم عبر `ToolRegistry` **دون معرفة تنفيذها**؛ الأداة محوّل فوق منافذ محقونة. أدوات الموصّلات/MCP تُحلّ ديناميكياً لكل Workspace عبر منفذ `integrations.ToolCatalog` (`FR‑52`) خلف نفس العقد.
- لا يستورد وكيلٌ وكيلاً آخر ولا API ولا infrastructure.

## 8.1) وحدتا `usage` و`integrations` (قواعد خاصّة)
- **فرض/التقاط `usage`** يُستدعى من **المُنسِّق (طبقة الوكلاء) حصراً** عبر المنافذ الواردة — لا من داخل وحدة أعمال أخرى؛ الفرض يعيد **كائن قرار** لا `bool` (`FR‑132`)، والالتقاط **متزامن idempotent بلا Redis Streams** (`FR‑131`).
- **أسرار `integrations`** (رموز OAuth) عبر `SecretsProvider` (Vault Transit) فقط — لا نصّ صريح، ولا تُعاد عبر أي استجابة (`SEC‑07`). **تجديد OAuth كسول** عند الاستخدام لا مجدوَل (`FR‑124`).
- **MCP بعيد (HTTP/SSE) حصراً**: **يُمنع** تشغيل خوادم MCP كعمليات فرعية داخل حاويات التطبيق/العمّال (`ARC‑15`).

## 9) الإعداد والأسرار
- **لا قراءة `.env` مباشرة** خارج `infrastructure/config`؛ الوصول عبر عقد `Settings` محقون.
- الأسرار عبر `SecretsProvider` فقط؛ لا سرّ في السجلّ أو الاستجابة أو الاستثناء.

## 10) التسجيل (Logging)
- Logging مُهيكل (JSON) يحمل `correlation_id` و`workspace_id` (نقطة توسعة Observability).
- لا يُسجَّل: توكنات، أسرار، مفاتيح API، محتوى مستخدم حسّاس.
- مستويات: `INFO` لأحداث الأعمال، `WARNING` للمتوقّع المتدهور، `ERROR` للفشل الحقيقي.

## 11) الالتزامات (Commits/PR)
- كل PR يمرّ البوّابات (بند 7 من `09-testing-strategy.md`).
- الهجرة تُراجَع ضدّ قاعدة «لا FK عابر بين schemas الوحدات».
- تغيير عقد منفذ/حدث/API ⇒ تحديث الوثيقة المقابلة في `docs/design/` في نفس الـPR.
